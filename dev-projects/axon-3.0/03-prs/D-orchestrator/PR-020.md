# PR-020 — Programs in REGISTRY

**Phase:** D — Orchestrator + UX
**Complexity:** M
**Status:** not-started
**Depends on:** none (most useful after PR-019)

## Carryover from Phase A/B execution

- **PR-008's deletions**: `save-workspace.md` and `code-dev-pr-spec.md` are gone. The new program registry must NOT include them. The `code-dev-pr-spec` legacy invocation now routes through `code-dev-pr --style opm` via dispatcher; if the registry lists invocation patterns, document this aliasing.
- **PR-006/7 deprecations**: `tools/REGISTRY.json` already marks `hooks`, `compile-write`, `compile-suggest`, `compile-optimizer` as OPTIONAL with deprecation notes. PR-020's program registry should mirror that pattern: programs deprecated this phase get a `status: DEPRECATED` field with a `successor` pointer (e.g., `save-workspace → workspace-backup push`).
- **Status enum extension**: `tools/REGISTRY.json` only knows ACTIVE/OPTIONAL/PLANNED/DEPRECATED today. Programs registry should use the same vocabulary so a single set of consumers can read both.

## Why

Today every program's metadata lives in its own header; menu/status/find-program scan all 66 files on every render. With 130+ programs predicted by year-end (library-dev growth + auto-compile in PR-017), this becomes a real cost. Registry is the standard fix.

## Implementation steps

1. **REGISTRY schema**:
```json
{
  "schema_version": 1,
  "generated_at": "2026-MM-DDTHH:MM:SSZ",
  "programs": {
    "menu": {
      "file": "workspace/programs/menu.md",
      "compiled": "workspace/programs/compiled/menu.cmp.md",
      "status": "ACTIVE",
      "area": "system",
      "description": "AXON home screen — OS state dashboard, mode switcher, active context",
      "tools": ["clock", "kv-store", "calculator"],
      "exec_count": 0,
      "last_modified": "2026-05-08T..."
    },
    ...
  }
}
```

2. **programs_registry.py**:
   - `programs-registry generate` — scan workspace/programs/*.md, extract headers, populate JSON
   - `programs-registry query --area code-dev` — return programs in that area
   - `programs-registry validate` — assert every .md has an entry; flag mismatches

3. **Update consumers** — menu.md, find-program.md, list-tools to read REGISTRY instead of scanning. Cuts boot time noticeably.

4. **Auto-regenerate** — cron entry (added by PR-011 seed-defaults extension):
```
{"id": "axon-programs-registry", "program": "programs-registry generate",
 "schedule": "weekly Mon 09:15", "label": "Refresh program registry"}
```

5. **Tests**:
   - `test_generate_covers_all_programs` — generate, assert each .md has entry
   - `test_query_by_area` — programs filter correctly
   - `test_validate_detects_mismatch` — delete an entry, validate, assert flagged

## Test strategy

- Pytest covers core mechanics.
- Benchmark: menu render time before/after; expect noticeable speedup with 80+ programs.

## Rollback plan

Delete REGISTRY.json + tool. Revert menu/find-program changes (they fall back to file scan).

## Acceptance criteria

- ✓ workspace/programs/REGISTRY.json exists with all program entries
- ✓ programs-registry tool functional
- ✓ Menu uses REGISTRY (verified via timing improvement)
- ✓ Cron entry registered for weekly regeneration
- ✓ Tests pass
- ✓ Health 100, lint clean

## Risks

- Registry can drift from filesystem (program added but registry not regenerated). Mitigation: weekly cron + boot-time staleness check (warn if oldest > 14d).
- Registry becomes a single point of failure. Mitigation: validate subcommand auto-runs at boot; on failure, fall back to file scan with WARN log.

## Out of scope

- Tool-style REGISTRY for programs (we're doing the analogous data, separate file).
- exec_count tracking from usage logs (auto-populated; not a hard requirement for v3.1).
- Cross-machine sync of exec_count (per-host).
