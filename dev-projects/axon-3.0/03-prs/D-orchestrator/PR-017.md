## Carryover from Phase A/B execution (significant)

Three changes from upstream PRs reshape this PR's auto-actions:

1. **Tool name change (PR-007)**: auto-compile must call `compile rank --auto` (not `compile-suggest suggest --auto`) and `compile format` (not `compile-write`). The legacy names still work via deprecation aliases for one release, but the orchestrator should use the canonical form. Update the action 1 spec from
   ```
   candidates ← TOOL(compile, rank, --window 7d --threshold 5 --uncompiled-only)
   ```
   ✓ already correct in this spec — was forward-looking.

2. **No real optimizer yet (PR-005 + PR-004)**: PR-005 wrote the spec for compiler optimization rules O1–O10, but **none are implemented** — the current compile pipeline is largely copy-with-header (compiled file ≈ source file). Implication: auto-compile produces .cmp.md files that:
   - Pass the structural regression test (PR-004's TestForwardCompilationCoverage)
   - **Do not actually save tokens** (test_compiled_token_ratio fails for these because the compiled file is the source plus a header)
   - Are still useful for dispatch (the registry index entry is what dispatch uses)
   The orchestrator should auto-compile based on usage threshold and not block on the token-ratio test. Audit log can record `optimization=none-yet` so future passes find them when the real optimizer ships.

3. **Pre-existing systemic issues that auto-compile must skip cleanly (PR-004 surfaced these)**:
   - **`!STUB` programs**: 4 library-dev programs (cite, intersect, report, search) are explicit Phase 2 placeholders. Auto-compile should detect the `!STUB` marker and skip them (recompiling a stub achieves nothing). Filter rule: skip programs whose source contains `!STUB`.
   - **`TOOL(shell)` not registered**: programs using `TOOL(shell, ...)` fail `verify.py` checks. Don't let a verify-failure on TOOL(shell) take down auto-compile — categorize separately and report rather than failing the run.
   - **3 pre-existing compiled programs (code-dev-study, resume, turn-log) lack the `# COMPILED:` header**. Don't let header-format checks block this either.

4. **Hook firing now reliable (PR-006)**: `fire_hook` was using cwd-relative `tools/log.py` paths and would fail when invoked outside the project root. PR-006 fixed it to use `AXON_ROOT`. The orchestrator can now reliably emit hooks from cron (which runs from the daemon's cwd, not the project root).

5. **Probe coverage (PR-003)**: All 6 orchestrator-critical tools (compile, dispatch, usage, events, igap, axon-audit) have functional probes — the orchestrator can `health-check` itself and trust the verdict.

## Implementation steps

1. **Create auto-improve.md program** with this structure:

```
PROGRAM: auto-improve
desc: Daily orchestrator. Closes auto-applicable improvement loops with full audit trail.

GUARD:
  ASSERT(L:auto-improve ≡ true) | DONE("auto-improve disabled")
  drift ← TOOL(drift, gate)
  ASSERT(drift.state ≠ "diverged") | FAIL(auto-improve, "drift state diverged; halting")
  IF drift.state ≡ "drifting" → confidence-modifier ← -30

LOAD CONTEXT:
  ts ← TOOL(clock)
  STORE(W:active-program, "auto-improve")
  STORE(W:active-phase, "auto-improve:start")

ACTION 1 — Auto-compile high-frequency programs:
  candidates ← TOOL(compile, rank, --window 7d --threshold 5 --uncompiled-only)
  ∀ c in candidates →
    snapshot ← TOOL(undo, snapshot --target workspace/programs/compiled/{c}.cmp.md)
    result ← TOOL(compile, optimize --program {c})
    IF result.ok → TOOL(auto-audit, record, --action compile --target {c} ...)
    ELSE → TOOL(auto-audit, record, --action compile-failed ...)

ACTION 2 — Auto-tune dispatch threshold:
  IF L:dispatch-auto-tune ≡ true →
    feedback ← TOOL(dispatch, feedback-summary, --window 20)
    IF feedback.neg-rate > 0.30 →
      old-threshold ← RETRIEVE(W:dispatch-confidence)
      new-threshold ← old-threshold + 0.05
      TOOL(undo, snapshot --target W:dispatch-confidence)
      STORE(W:dispatch-confidence, new-threshold)
      TOOL(auto-audit, record, --action tune-threshold --before {old} --after {new})

ACTION 3 — Auto-archive episodic memory:
  TOOL(memory-compact, --age 30d --auto)
  IF compact.archived > 0 → TOOL(auto-audit, record, --action archive ...)

OUTPUT:
  → "Auto-improvements: {count} action(s) taken"
  STORE(W:active-phase, "auto-improve:done")
  CHECKPOINT
DONE(auto-improve)
```

## Risks

- **Major risk: orchestrator runs amok**. Mitigation: default off; gated by drift; only 3 narrow actions; everything audited + rollback-able.
- Cron daemon might run auto-improve at the same time as user is doing something. Mitigation: rate limit per PR-010 (1 cron job per tick).
- Auto-compile breaks a hot path. Mitigation: rollback snapshot taken before; user can revert via undo.
- Auto-tune over-tightens dispatch threshold. Mitigation: caps at 0.95; user can manually re-tune.

## Out of scope (intentionally)

- Auto-editing kernel/LANG/program content (forbidden by Core Rule 9 + 10).
- Auto-applying igap-improve fixes (those involve content edits; stays human-confirmed).
- Auto-adding new tools or extending LANG (EXTEND protocol stays manual).
- Cross-program optimization (e.g. merging two programs that look similar). Single-program scope only.
- Multi-user / multi-host coordination.
