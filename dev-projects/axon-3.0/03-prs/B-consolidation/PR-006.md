# PR-006 — events + hooks unification

**Phase:** B — Tool consolidation
**Complexity:** M
**Status:** completed
**Depends on:** PR-002 (canonical envelope, atomic writes)

## Spec deviation (recorded post-execution)

Original spec proposed `events hook --add`/`--remove`/etc. as a single sub-action with flags. **Executed** as flat top-level actions instead: `hook-add`, `hook-list`, `hook-remove`, `hook-fire`, `hook-enable`, `hook-disable`. Reasons:
- Simpler argparse (no nested subparsers).
- Matches the existing flat action style of every other AXON tool.
- Cleaner migration mapping: `hooks add` → `events hook-add` (1:1 rename).

**Bonus fix:** while migrating `fire_hook`, anchored the subprocess paths to `AXON_ROOT` (was `tools/log.py` cwd-relative). Pre-existing bug from `hooks.py` — fire would fail when invoked outside the project root. Now works from any cwd.

## Why

Conceptually one system: events emit, hooks listen. Two tools with overlapping config invites drift. Saves one tool and simplifies the orchestrator's signal-routing in Phase D.

## Implementation steps

1. **Map current hooks.py surface**:
   - `hooks add --event X --action Y --target Z`
   - `hooks remove --id N`
   - `hooks list`
   - `hooks fire --event X` (manual trigger?)

2. **Merge into events.py**:
   - `events emit --event X --payload {...}` (existing)
   - `events listen --since T --filter X` (existing)
   - `events hook --add --event X --action Y --target Z`
   - `events hook --remove --id N`
   - `events hook --list`
   - `events hook --fire --event X` (test fire)

3. **Migrate existing hook configs** — read `workspace/memory/longterm/hooks.json` (if exists), import each into events tool's config (`workspace/memory/longterm/events-hooks.json`). On migration: log entries written, keep old file with `.migrated` suffix.

4. **Update programs** — grep `workspace/programs/` for `TOOL(hooks` and `TOOL(events`. Migrate `TOOL(hooks, ...)` to `TOOL(events, hook ...)`.

5. **Update menu/help** — `tools/list_tools` and `axon-docs-gen` should reflect the consolidation.

6. **Alias compat** — `axon.py:ALIASES["hooks"] = "events"` so old programs still work during transition. One-release deprecation window.

7. **Tests** — `TestEvents::test_hook_add_remove_list_roundtrip`, `test_hook_fire_event_listened`.

## Test strategy

- Roundtrip: add hook, fire its event, verify hook ran (proxy: writes a flag file).
- Backward compat: invoking `axon.py hooks add ...` still works via alias.
- All program calls migrate cleanly: grep after migration shows zero `TOOL(hooks` in workspace/programs/.

## Rollback plan

If migration breaks user hooks, revert events.py and re-instate hooks.py from git. The migration script makes a backup (`hooks.json.migrated`) so config is recoverable.

## Acceptance criteria

- ✓ `tools/hooks.py` deleted
- ✓ `events.py` exposes hook subcommands
- ✓ `axon.py hooks ...` still works via alias
- ✓ All workspace programs migrated; grep shows clean
- ✓ TestEvents::test_hook_* pass
- ✓ Health 100, lint clean

## Risks

- A user has authored custom hooks in workspace/preferences/ that aren't in hooks.json. Mitigation: the migration step logs everything imported; user can audit.
- Programs that pass complex JSON to `TOOL(hooks, add, --action ...)` may have escape-character issues in subcommand form. Mitigation: subcommand parser uses argparse correctly.

## Out of scope

- Removing the `hooks` alias (deferred one release).
- Changing the event schema (separate concern).
