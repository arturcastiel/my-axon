**Complexity:** M
**Status:** completed
**Depends on:** PR-002

## Spec deviation (recorded post-execution)

**Threshold reconciliation**: spec proposed 0.3 / 0.6 thresholds (drifting/diverged); the existing `drift.py:classify()` already used 0.10 / 0.40. Used the existing thresholds — duplicating bands risked the gate's `state` field disagreeing with `drift check`'s `status` field. The integer constants `GATE_DRIFT_HALT = 0.40` and `GATE_DRIFT_WARN = 0.10` are top-of-module so future tuning is a single edit.

**Rule-based wiring instead of bespoke verify integration**: spec said "wire into response gate via verify.py call". Implemented as a new pluggable rule (`tools/rules/r_drift_gate.py`) that registers alongside the existing 8 kernel rules. Cleaner — drift halt becomes one of N rules verify.py already runs, no special-casing.

**Dev-mode demotion**: when `L:dev-mode ≡ true`, drift halt is demoted to WARN instead of being silently ignored. Aligns with the kernel's strict/soft halt-mode pattern. Under strict halt-mode the demoted WARN still fails verify (passed=false); under soft halt-mode it surfaces but allows render. User explicitly chooses both levers.

**Output-layer footer wiring deferred**: original spec called for footer to show drift status. Menu badge added (`workspace/programs/menu.md` shows drift state when not stable); footer mechanic is more invasive (touches axon/OUTPUT-LAYER.md under dev-mode) and the menu surface is the visible signal where users actually look. PR-018 (UX panel) will absorb the footer change as part of its self-improvement-panel work.

## Why

Drift is computed every turn and read by zero decision logic. Either it matters (and should affect behavior) or it should be removed. This PR makes it matter: high drift halts; medium drift reduces confidence; low is quiet.

## Implementation steps

1. **drift.py gate subcommand**:
   ```
   drift gate [--workspace W:active-program]
   ```
   Reads current program's drift score. Returns:
   ```json
   {"ok": true, "data": {"score": 0.42, "state": "drifting", "decision": "warn", "modifier": -30}}
   ```
   States:
   - `stable` (score < 0.3) → decision `quiet`, modifier 0
   - `drifting` (0.3 ≤ score ≤ 0.6) → decision `warn`, modifier -30 (confidence penalty)
   - `diverged` (score > 0.6) → decision `halt`, modifier -50

2. **Wire into response gate** — `verify.py` calls `drift gate` (if active program known); if decision == `halt`, response-gate fails with "drift exceeded threshold for {program}; rerun is recommended". User can override via dev-mode.

3. **Output layer** — `axon/OUTPUT-LAYER.md` (dev-mode edit):
   - In compact format: `[drift: stable|drifting|diverged]` badge inline with confidence.
   - Add explanation in OUTPUT-LAYER.md that drift now affects confidence.

4. **Menu badge** — `workspace/programs/menu.md` OS STATE panel adds a "Drift" line:
   ```
   Drift         ●●●○○ drifting (score 0.42)  ⚠ run: drift check {program}
   ```
   Only shown if not stable.

5. **Kernel rule** (dev-mode) — KERNEL-SLIM:
   - New COMPLIANCE entry "Drift gate" — fires before render of any program output. Decision `halt` → FAIL block. Decision `warn` → confidence penalty.

6. **Tests**:
   - `test_gate_returns_state` — known score → expected decision/modifier
   - `test_response_gate_halts_on_diverged` — synthesize a diverged program, run, assert FAIL
   - `test_warn_reduces_confidence` — drift = 0.45, response confidence drops by 30

## Test strategy

- Synthesize known drift score by populating drift state file in a tmp_ws.
- End-to-end: run a program that diverges from its expected tool sequence, observe halt.

## Rollback plan

Revert drift.py + verify.py changes. KERNEL-SLIM rule revert (dev-mode). Menu / output-layer edits revert independently.

## Acceptance criteria

- ✓ `drift gate` returns structured decision
- ✓ Response gate uses drift in its calc
- ✓ High drift halts; medium reduces confidence
- ✓ Drift visible in menu + output layer footer
- ✓ Tests pass
- ✓ Health 100, lint clean

## Risks

- A noisy drift signal halts useful work. Mitigation: thresholds are conservative (0.6 for halt); user can override via dev-mode.
- Drift only meaningful for programs with `expected tool sequence` declared; many programs don't have one. Mitigation: if no expected sequence, decision = `quiet`.

## Out of scope

- Auto-recompiling diverged programs (Phase D / PR-017).
- Bulk drift initialization for all programs (PR-018 menu surfaces this).
