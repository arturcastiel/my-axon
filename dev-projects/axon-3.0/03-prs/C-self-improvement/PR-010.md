- 3-letter day names (`Mon`/`mon`/`MON`) via new `DAYS_SHORT` map and `_resolve_day()`
- `monthly D HH:MM` format (D = 1..28 for portability — Feb 30 isn't a thing)

## Why

Cron has been registered since 2026-05-03 (`run_count=0`). Without an active scheduler firing entries, every Phase C/D autonomy plan stays manual. This is the smallest change that unblocks the whole self-improvement story.

AXON has no daemon process; "active scheduler" here means a boot-time tick that surfaces overdue jobs and either runs them automatically (opt-in) or queues them with a one-line approve prompt.

## Implementation steps

1. **Add `cron tick`** to `tools/cron.py`:
   - Read cron.json
   - For each entry with schedule due (compare against `last_run_ts`):
     - If `L:cron-auto ≡ true`: run the program/command. On success, update `last_run_ts` + increment `run_count`. Rate limit: at most 1 auto-run per tick (rest queue).
     - If `L:cron-auto ≡ false` (default): append to a "pending" list, return it.
   - Output: `{"ran": [...], "pending": [...], "errors": [...]}`

2. **Wire into boot.py** — after kernel internalize, before menu, call `cron tick`. Surface pending list in boot trace:
   ```
   ⏰ Cron — N pending jobs (igap report, compile rank, ...)
      Run them with: cron tick --auto
   ```

3. **Add preference** — `workspace/preferences/agent.md` adds:
   ```
   cron-auto: false
   # When true, boot-time cron tick runs overdue jobs automatically.
   # Default false; set true via `kv-store set --key L:cron-auto --value true`
   # once trust is established with audit trail (PR-015).
   ```

4. **Update kernel** (dev-mode) — KERNEL-SLIM BOOT STEP 3 mentions:
   - "Cron tick: overdue jobs surfaced; run automatically if L:cron-auto ≡ true."

5. **Tests**:
   - `test_tick_lists_overdue`: register a cron with past last_run_ts, tick, assert in `pending`
   - `test_tick_auto_runs_when_enabled`: set L:cron-auto=true, tick, assert ran + run_count incremented
   - `test_tick_rate_limits`: register 5 overdue jobs with auto on, tick, assert only 1 ran (4 still pending)
   - `test_tick_records_errors`: register a cron pointing to nonexistent program, tick, assert errors list populated and run_count NOT incremented

## Test strategy

- Pytest fixture: tmp_ws with crafted cron.json + mocked subprocess invocations of programs.
- End-to-end: register a real cron entry, advance fake clock, boot, assert tick executes.

## Acceptance criteria

## Risks

- A misregistered cron entry could spam errors on every boot. Mitigation: errors list visible; user can `cron remove` problematic entries.
- Boot becomes slower if many overdue jobs. Mitigation: rate limit at 1; rest queue.
- Auto-running long jobs blocks the boot trace. Mitigation: jobs intended for the orchestrator are short-running; long jobs stay in queue + manually invoked.

## Out of scope

- Default cron entries (PR-011).
- Background daemon process (out of scope for now; boot-tick is enough).
