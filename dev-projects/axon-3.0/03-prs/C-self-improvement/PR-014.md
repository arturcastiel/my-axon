## Why

## Implementation steps

1. **Implicit positive feedback** — after a dispatch match, record the dispatch ID and timestamp in `W:_last-dispatch`. If the next user input is a continuation command (per kernel line 181: yes/y/continue/etc.), emit `dispatch feedback --id <id> --result yes`.

2. **Implicit negative feedback** — when igap.py records an `absent-instruction` event, check `W:_last-dispatch`. If recorded ≤2 turns ago, emit `dispatch feedback --id <id> --result no`. Also: if user re-states a similar prompt within 2 turns of the dispatch, infer negative.

3. **Auto-tune toggle** — `L:dispatch-auto-tune ≡ true` (default false). Threshold-tuning logic in dispatch.py reads this before adjusting.

4. **Reset window** — Phase 1 study showed feedback log empty. After implicit feedback works for 20 dispatches, threshold may auto-adjust per existing logic.

5. **Tests**:
   - `test_continuation_records_positive` — match, then continuation, assert feedback log has entry
   - `test_igap_correlation_records_negative` — match, then igap absent-instruction, assert negative
   - `test_auto_tune_disabled_default` — accumulate 20 negatives with toggle off, threshold unchanged
   - `test_auto_tune_tightens_when_enabled` — with toggle on, threshold rises after sufficient negatives

## Acceptance criteria

- ✓ Continuation commands record positive feedback

## Risks

- False positives: user typing "yes" might not actually be confirming the dispatch, just continuing a separate program. Mitigation: only record when `W:_last-dispatch` is recent and current program matches.
- False negatives: igap might fire for unrelated reasons. Mitigation: same recency + program check.

## Out of scope

- Adjusting other dispatch parameters beyond threshold (e.g. semantic vs lexical weight).
- Per-program threshold tuning (start global, refine later if needed).
