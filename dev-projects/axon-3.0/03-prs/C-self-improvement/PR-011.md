# PR-011 — Default self-improvement cron schedule

**Phase:** C — Self-improvement plumbing
**Complexity:** XS (most work folded into PR-010)
**Status:** completed
**Depends on:** PR-010

## Carryover from PR-010 execution

PR-010 ended up shipping `cron seed-defaults` (the heart of this PR) because the tick action it built was meaningless without entries to fire. The 5 default cron entries are already added by `seed-defaults`:
- `axon-igap-report` weekly Mon 09:00
- `axon-compile-rank` weekly Mon 09:05
- `axon-dispatch-stats` weekly Mon 09:10
- `axon-memory-compact` monthly 1 09:00
- `axon-session-save` daily 23:00

The action is idempotent and tested.

**What's actually left for PR-011**:
1. **First-boot autowire**: have `tools/boot.py` call `cron seed-defaults` once when the workspace shows zero entries, gated by `L:axon-cron-defaults-seeded` (set true after first invocation). Currently seed-defaults is a CLI command users have to invoke manually.
2. **Optional schedule tuning**: re-evaluate the 5 default cadences based on real-world experience and adjust if needed.

PR-011 is now a small wire-up PR. It can land any time — no longer blocking.

## Spec deviation (recorded post-execution)

**Sentinel-based gating** — chose a filesystem flag (`<ws>/memory/kv-store/axon-cron-defaults-seeded.md`) over a runtime L: key. Reasons:
- The flag persists naturally across sessions without needing kv-store wiring at boot time.
- A user who deletes a default cron entry shouldn't see it re-seeded on next boot. The sentinel gives that contract: "we seeded once; never again on this workspace".
- Easy to inspect (it's just a file).

**As executed**:
- ✓ `tools/boot.py`: new `seed_cron_if_first_boot()` — checks for the sentinel; if absent, calls `cron seed-defaults`, writes the sentinel on success. Sequenced BEFORE `check_cron`/`tick_cron` so the same boot can both seed AND fire if `L:cron-auto ≡ true`.
- ✓ Boot JSON `cron.seeded` field — populated only on the first boot (the seed result), `null` thereafter.
- ✓ Tests: `TestCronFirstBootAutowire` — 3 cases (first-boot seeds 5, second boot doesn't reseed, user-deleted entry stays deleted).
- ✓ Schedule cadence tuning deferred — current defaults are reasonable; revisit only if real usage demands.

PR-011 footprint vs PR-010: ~15 lines of boot.py + 3 tests. Tiny.

## Why

Without these defaults, PR-010's daemon has nothing meaningful to run. The 5 scheduled jobs match exactly what the system was designed for — their tools exist, their consumers are wired, they were just never scheduled.

## Implementation steps

1. **Add `cron seed-defaults`** to `tools/cron.py`:
   - Idempotent — uses `--if-absent` semantics for each entry. If a job with the same `id` exists, skip.
   - Adds 5 default entries:
     ```json
     [
       {"id": "axon-igap-report", "program": "igap report --days 7",
        "schedule": "weekly Mon 09:00", "label": "Weekly inference gap report"},
       {"id": "axon-compile-rank", "program": "compile rank --window 7d",
        "schedule": "weekly Mon 09:05", "label": "Weekly compile candidates"},
       {"id": "axon-dispatch-stats", "program": "dispatch-stats weekly",
        "schedule": "weekly Mon 09:10", "label": "Weekly dispatch summary"},
       {"id": "axon-memory-compact", "program": "memory-compact --age 30d",
        "schedule": "monthly 1 09:00", "label": "Monthly episodic compaction"},
       {"id": "axon-session-save", "program": "session-save",
        "schedule": "daily 23:00", "label": "Daily session snapshot"}
     ]
     ```

2. **First-boot detection** — boot.py checks `L:axon-cron-defaults-seeded`. If absent, call `cron seed-defaults`, then set `L:axon-cron-defaults-seeded = true`.

3. **Tests**:
   - `test_seed_defaults_creates_5_entries` — empty cron.json, run seed-defaults, assert 5 entries.
   - `test_seed_defaults_idempotent` — run twice, assert still 5 entries (no duplicates).
   - `test_seed_defaults_respects_user_entries` — manually add a job with same id, run seed-defaults, assert user job preserved.

## Test strategy

- Pytest covers idempotency.
- Smoke: run boot in a fresh workspace, verify `cron list` shows 5 default entries on second boot (first boot just seeds them; second boot's tick may surface them as pending).

## Rollback plan

Revert cron.py changes. Existing cron.json entries keep working; new entries can be removed with `cron remove --id axon-...`.

## Acceptance criteria

- ✓ `cron seed-defaults` adds 5 entries idempotently
- ✓ Boot triggers seed on first run only
- ✓ User-defined cron entries are preserved
- ✓ Test suite passes
- ✓ Health 100, lint clean

## Risks

- Schedule strings must be parseable by cron.py; verify each format works (`weekly Mon 09:00`, `monthly 1 09:00`, `daily 23:00`).
- If a referenced tool doesn't exist (e.g. `dispatch-stats weekly` is a CLI form), cron tick will error on that entry. Mitigation: run each command manually first to confirm syntax.

## Out of scope

- Tuning the schedule cadences (start with the suggested defaults; adjust later based on experience).
- Adding entries for things not yet built (the orchestrator entry comes in PR-017).
