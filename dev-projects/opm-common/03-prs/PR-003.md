## What This PR Does

`Summary.cpp` contains a `Factory::create()` function that dispatches each
`SummaryNode` to an evaluator class. When the simulator produces a value for a
summary vector every report step, this factory determines *how* to extract that
value. Currently, the dispatch chain has no arm for LGR nodes, so any node whose
`lgr` field has a value falls through to `unknownParameter()` — which creates no
evaluator — and the LW* vectors are silently omitted from every UNSMRY file.

This PR adds a new evaluator class `LgrWellValue : public Base` and a
corresponding `Factory::isLgrWellValue()` guard that routes LGR well nodes before
they reach `isFunctionRelation()`. The evaluator resolves the base keyword
(`LWOPR` → `WOPR`, `LWBHP` → `WBHP`) and delegates to the same `funs` map entry
that the equivalent global well keyword uses, after verifying that the named well
is actually within the named LGR.

The change is purely additive: no existing evaluator is modified, the existing
dispatch order is preserved, and no non-LGR vector is affected.

**Trap:** `isBlockValue()` (line ~4837) calls `this->grid_.cellActive(this->node_->number - 1)` which is `cellActive(-1)` when `number == 0`. This would be UB. The new `isLgrWellValue()` guard must fire **before** `isBlockValue()`, or at minimum before `isFunctionRelation()`. The safe position is after `isGlobalProcessValue()` because global process nodes have `number == 0` too (no cell) and are already handled before the block check.

Actually re-reading the dispatch order in `Factory::create()` (line ~4714):
```
isUserDefined()       → first (UDQ)
isBlockValue()        → second ← BUG RISK for LGR nodes if they have number=0
```
However the current `isBlockValue()` checks `block_units.find(node->keyword)` first —
`LWOPR` is not in `block_units`, so the map lookup fails before the `cellActive` call.
The LGR node safely falls through today without crashing. But to be explicit and safe,
insert `isLgrWellValue()` immediately after `isGlobalProcessValue()`.

## Fix 1: Add `lgr_well_units` map

### File: `opm/output/eclipse/Summary.cpp`

### Why This Map Exists
`Factory::isLgrWellValue()` must both (a) detect that a node is an LGR well vector
and (b) capture the unit of measure needed to convert the SI value to output units.
The existing `block_units` and `single_values_units` maps do the same thing for block
and global vectors. A dedicated `lgr_well_units` map follows the same pattern.

The map is keyed on the full LW* keyword name (not the stripped base). This is the
same string that appears in `node_->keyword`. The value is the `measure` enum entry.

### New Code (add near the other `*_units` maps, ~line 220)
```cpp
static const std::unordered_map<std::string, Opm::UnitSystem::measure>
lgr_well_units = {
    { "LWBHP",  Opm::UnitSystem::measure::pressure           },
    { "LWTHP",  Opm::UnitSystem::measure::pressure           },
    { "LWOPR",  Opm::UnitSystem::measure::liquid_surface_rate },
    { "LWWPR",  Opm::UnitSystem::measure::liquid_surface_rate },
    { "LWGPR",  Opm::UnitSystem::measure::gas_surface_rate    },
    { "LWWIR",  Opm::UnitSystem::measure::liquid_surface_rate },
    { "LWGIR",  Opm::UnitSystem::measure::gas_surface_rate    },
    { "LWOPT",  Opm::UnitSystem::measure::liquid_surface_volume },
    { "LWWPT",  Opm::UnitSystem::measure::liquid_surface_volume },
    { "LWGPT",  Opm::UnitSystem::measure::gas_surface_volume    },
    { "LWWIT",  Opm::UnitSystem::measure::liquid_surface_volume },
    { "LWGIT",  Opm::UnitSystem::measure::gas_surface_volume    },

    { "LWLPR",  Opm::UnitSystem::measure::liquid_surface_rate   },
    { "LWLPT",  Opm::UnitSystem::measure::liquid_surface_volume },

```

**Why `measure::liquid_surface_rate` for LWOPR/LWWPR:**
The `funs` map entries for `WOPR`/`WWPR` return `measure::liquid_surface_rate`.
The LGR evaluator calls the same `ofun` and must use the same unit to get the
same conversion factor. Using a different measure would silently produce wrong
numbers.

## Fix 2: Add `base_keyword()` file-local helper

### File: `opm/output/eclipse/Summary.cpp`

### Why This Function Exists
`LWOPR` must map to the `WOPR` entry in `funs`. The only difference is the
leading `L`. A one-line helper avoids duplicating this logic in every call site.

Per masterplan: keep this as a file-local static in `Summary.cpp`. Do **not** add
it to `SummaryNode.hpp` — it is an implementation detail of the evaluator layer.

### New Code (add as file-local static function, near `lgr_well_units`)
```cpp
static std::string base_keyword(const std::string& lgr_kw)
{
    // Strip leading 'L' from LW*/LC*/LB* keywords: "LWOPR" → "WOPR"
    return (lgr_kw.size() > 1 && lgr_kw[0] == 'L')
        ? lgr_kw.substr(1)
        : lgr_kw;
}
```

## Fix 3: Add `LgrWellValue` evaluator class

### File: `opm/output/eclipse/Summary.cpp`

### Why This Class Exists
All evaluator classes follow the `Base` interface (single `update()` virtual).
`LgrWellValue` must:
1. Confirm the well named in `node_.wgname` is actually within the LGR named in
   `node_.lgr->name` (via `Well::get_lgr_well_tag()`).
2. Delegate the actual rate/pressure extraction to the existing `ofun` from `funs`.
3. Write the result via `updateValue()`.

Skipping check (1) would produce wrong output when two wells share a name with
different LGR assignments — a case the masterplan explicitly calls out for
`LWBHP` in the wrong-LGR test.

### New Code (add after `GlobalProcessValue` class, before `UserDefinedValue`, ~line 4530)
```cpp
class LgrWellValue : public Base
{
public:
    explicit LgrWellValue(Opm::EclIO::SummaryNode     node,
                          const Opm::UnitSystem::measure m,
                          ofun                          fcn)
        : node_(std::move(node))
        , m_   (m)
        , fcn_ (std::move(fcn))
    {}

    void update(const std::size_t       sim_step,
                const double            stepSize,
                const InputData&        input,
                const SimulatorResults& simRes,
                Opm::SummaryState&      st) const override
    {
        // Verify that the well is actually in the specified LGR.
        // get_lgr_well_tag() returns nullopt for WELSPECS wells and the
        // LGR name for WELSPECL wells.  Both an absent well and a well in
        // a different LGR produce no output (silent zero / no-op).
        const auto& well_name = this->node_.wgname;
        const auto& lgr_name  = this->node_.lgr->name;

        if (!input.sched.hasWell(well_name, sim_step)) {
            return;
        }

        const auto& well = input.sched.getWell(well_name, sim_step);
        const auto  tag  = well.get_lgr_well_tag();
        if (!tag.has_value() || *tag != lgr_name) {
            return;
        }

        const auto wells = find_wells(input.sched, this->node_,
                                      static_cast<int>(sim_step), input.reg);

        const fn_args args {
            wells, std::string{""}, this->node_.keyword,
            stepSize, static_cast<int>(sim_step),
            /*number=*/0, this->node_.fip_region,
            st,
            simRes.wellSol, simRes.wbp, simRes.grpNwrkSol,
            input.reg, input.grid, input.sched,
            /*efac=*/{},
            input.initial_inplace, simRes.inplace,
            input.sched.getUnits(),
            simRes.rc_rates
        };

        const auto& usys = input.es.getUnits();
        const auto  prm  = this->fcn_(args);

        updateValue(this->node_, usys.from_si(prm.unit, prm.value), st);
    }

private:
    Opm::EclIO::SummaryNode  node_;
    Opm::UnitSystem::measure m_;
    ofun                     fcn_;
};
```

**Why call `find_wells()` inside `update()`:**
`FunctionRelation::update()` (line ~4132) calls `find_wells()` to build the
`wells` vector passed to `fn_args`. The `ofun` lambdas in `funs` — such as
`rate<rt::oil, producer>` — iterate this vector to sum rates. Without it, the
vector is empty and every rate returns zero. LGR well vectors need the same
mechanism: the well IS in `simRes.wellSol`, and `find_wells()` locates it via
the schedule.

**Why `/*number=*/0`:**
LW* nodes have `node_.number == 0` (no cell index). The `fn_args.number` field
is used only by connection and block lambdas, not by well-rate lambdas. Passing
0 is correct.

**Why no `EfficiencyFactor`:**
Rate vectors (LWOPR, LWWPR, LWGPR) are not accumulated (not Type::Total), so
efficiency factor does not apply — matching the comment in `FunctionRelation` which
only applies eFac to accumulated values. This is consistent with how WOPR is
evaluated.

## Fix 4: Add `Factory::isLgrWellValue()` and `Factory::lgrWellValue()`

### File: `opm/output/eclipse/Summary.cpp`

### New Methods (in the `Factory` private section, ~line 4700)
```cpp
bool Factory::isLgrWellValue()
{
    // Must have an LGR field and be a Well category node.
    if (!this->node_->lgr.has_value()) return false;
    if (this->node_->category != Opm::EclIO::SummaryNode::Category::Well)
        return false;

    auto pos = lgr_well_units.find(this->node_->keyword);
    if (pos == lgr_well_units.end()) return false;

    // Capture unit and the base-keyword function for delegation.
    this->paramUnit_ = pos->second;

    const auto normKw = Opm::EclIO::SummaryNode::normalise_keyword(
        Opm::EclIO::SummaryNode::Category::Well,
        base_keyword(this->node_->keyword));

    auto fpos = funs.find(normKw);
    if (fpos == funs.end()) return false;

    this->paramFunction_ = fpos->second;
    return true;
}

Factory::Descriptor Factory::lgrWellValue()
{
    auto desc = this->unknownParameter();
    desc.unit = this->directUnitString();
    desc.evaluator.reset(new LgrWellValue {
        *this->node_, this->paramUnit_, std::move(this->paramFunction_)
    });
    return desc;
}
```

Add declarations to the `Factory` private section in the class definition:

### Fix 5: Insert dispatch arm in `Factory::create()`

### File: `opm/output/eclipse/Summary.cpp`

### Current Code (~line 4714)
```cpp
    if (this->isGlobalProcessValue())
        return this->globalProcessValue();

    if (this->isFunctionRelation())
        return this->functionRelation();
```

### Change
```cpp
    if (this->isGlobalProcessValue())
        return this->globalProcessValue();

    if (this->isLgrWellValue())
        return this->lgrWellValue();

    if (this->isFunctionRelation())
        return this->functionRelation();
```

**Why after `isGlobalProcessValue()` and before `isFunctionRelation()`:**
- `isFunctionRelation()` looks up `normalise_keyword(category, keyword)` in `funs`.
  `LWOPR` normalises to `"LWOPR"` (unknown prefix), not `"WOPR"`, so it would
  return false anyway — but intercepting before that is explicit and O(1).
- `isBlockValue()` and `isAquiferValue()` are safe to run first (map lookups fail
  immediately for LW* keywords), but inserting before `isFunctionRelation()` places
  the LGR arm at the correct semantic tier: "structured well value with LGR identity".

## What This PR Does NOT Do

- **LC* connection evaluators** — deferred to PR-004. Same pattern; different
  `node.category == Category::Connection` guard.
- **LB* block evaluators** — deferred to opm-simulators (PR6). Requires
  `EclipseGrid::getActiveIndexLGR()` from `EclipseState` not available in opm-common.
- **Integration tests** (`LWOPR == WOPR` with a full parsed deck) — deferred to PR-005.
- **`LWOPT` / `LWWPT` / `LWGPT` accumulated totals** — listed in `lgr_well_units`
  but require `stepSize * rate` handling. The `funs` map entries `WOPT`/`WWPT`/`WGPT`
  already handle this via `mul(rate<...>, duration)`. LgrWellValue delegates to the
  same `ofun`, so totals should work automatically. Verify in PR5.
- **`ROOT.LGR`** file creation — permanently out of scope.
- **Efficiency factor for LW* accumulated totals** — `FunctionRelation` applies
  eFac for accumulated values. For PR3, omit eFac (matches rate-only usage).
  Revisit if accumulated LW* vectors are needed.
