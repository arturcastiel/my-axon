## Design Decision — `lgr_name_` not `lgr_info`

The config layer stores only the LGR **name** (`optional<string> lgr_name_`), not the full `lgr_info` struct with ijk coordinates. This is a deliberate layering decision:

- **ijk belongs in the IO layer.** `lgr_info.ijk` is populated from the `NUMLX/NUMLY/NUMLZ` arrays in the binary SMSPEC file — a concern of `ESmry.cpp`, not the config parser.
- **The config layer has no ijk at parse time** for LW* (wildcard) and may not have it reliably for LC* until connection data is resolved.
- **`operator<` ordering** on ijk via `std::array::operator<` is lexicographic (i.e. `{1,2,3} < {2,2,2}`), not declaration order — an ordering bug flagged in review.

The conversion operator bridges the two layers:
```cpp
// Config layer (lgr_name_ = "LGR1") → IO layer (lgr_info{name, {}})
/* lgr = */ this->lgr_name_.has_value()
            ? std::optional<EclIO::lgr_info>{ EclIO::lgr_info{ *this->lgr_name_, {} } }
            : std::optional<EclIO::lgr_info>{}
```
The `{}` ijk placeholder is intentional. PR-002 fills real ijk from SMSPEC arrays.

## Fix 1 — `SummaryNode.cpp`: `case 'L':` in `category_from_keyword()`

**Problem:** `category_from_keyword()` had no `case 'L':` — any `LW*`, `LC*`, `LB*` keyword fell through to `default` and returned `Category::Miscellaneous`. A vector classified as Miscellaneous is never treated as a Well, Connection, or Block — no well or cell data is ever looked up. No LGR output would ever be produced without this fix.

**Change:** Inserted between `case 'G':` and `case 'R':` (preserving alphabetical order):

```cpp
case 'L':
    if (keyword.size() >= 2) {
        switch (keyword[1]) {
        case 'W': return distinguish_well_from_completion(keyword);
        case 'C': return distinguish_connection_from_completion(keyword);
        case 'B': return Category::Block;
        default:  break;
        }
    }
    return Category::Miscellaneous;
```

`distinguish_well_from_completion` and `distinguish_connection_from_completion` are file-local helpers already defined in `SummaryNode.cpp` — reused here for `LW*`/`LC*` exactly as for `W*`/`C*`. `LLINEARS` (`keyword[1]='L'`) hits `default: break` and returns `Miscellaneous` — correct, it is a solver statistic.

## Fix 2 — `SummaryConfig.hpp`: `lgr_name_` field, getter, setter, conversion, serialize

### The `lgr_name_` private field

```cpp
/// LGR name for LGR-level summary vectors (LW*, LC*, LB*).
/// std::nullopt for all non-LGR vectors (the overwhelming majority).
std::optional<std::string> lgr_name_{};
```

Added after `KeywordLocation loc{}` in the private section.

### The getter

```cpp
const std::optional<std::string>& lgr_name() const
{ return this->lgr_name_; }
```

### The setter

```cpp
SummaryConfigNode& lgr_name(const std::string& name)
{ this->lgr_name_ = name; return *this; }
```

PR-001's `keywordLW`, `keywordLC`, `keywordLB` use the existing constructor and call the setter immediately after:

```cpp
// PR-001 — keywordLW (simplified)
auto node = SummaryConfigNode("LWOPR", Category::Well, kw.location());
node.namedEntity(well_name).lgr_name(lgr_name);
list.push_back(node);
```

### Fix 2c — conversion operator `EclIO::SummaryNode()`

```cpp
// Before:
/* lgr = */ {}   // always null

// After:
auto lgr = this->lgr_name_.has_value()
    ? std::optional<EclIO::lgr_info>{ EclIO::lgr_info{ *this->lgr_name_, {} } }
    : std::optional<EclIO::lgr_info>{};
return { keyword_, category_, type_, name_, number_, fip_region_, std::move(lgr) };
```

This is the gate through which LGR identity reaches `OutputStream` (PR-002) and `Summary.cpp` (PR-003/004).

### Fix 2d — `serializeOp()`: append `serializer(lgr_name_)` last

```cpp
serializer(userDefined_);
serializer(lgr_name_);    // ← added, MUST be last
```

Must be last to preserve binary compatibility with existing restart files. `optional<string>` is natively handled by the OPM serializer — no additional `serializeOp` on `lgr_info` is needed, and `SummaryNode.hpp` is therefore left unchanged.

## Fix 3 — `SummaryConfig.cpp`: `operator==` and `operator<`

**Problem:** Before PR-000, neither comparator included LGR identity. `LWOPR+LGR1+PROD1` and `LWOPR+LGR2+PROD1` compared equal — same keyword, same name — so one node was silently dropped from the `std::set`. No error, just missing output.

### `operator==` — added to all three category groups

```cpp
// Well / Group / Node case:
return (lhs.namedEntity() == rhs.namedEntity())
    && (lhs.lgr_name()    == rhs.lgr_name());   // ← added
```

### `operator<` — full tiebreaker chain in all three groups

```cpp
// Well case (example):
if (lhs.namedEntity() != rhs.namedEntity())
    return lhs.namedEntity() < rhs.namedEntity();
if (lhs.lgr_name().has_value() != rhs.lgr_name().has_value())
    return rhs.lgr_name().has_value();   // no-lgr sorts before lgr
if (!lhs.lgr_name().has_value())
    return false;                        // both null → equal
return *lhs.lgr_name() < *rhs.lgr_name();
```
