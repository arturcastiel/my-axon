## Mental Model

Before this PR, the LGR summary pipeline has a gap at the evaluator layer. The `Factory::create()` dispatch chain in `Summary.cpp` determines — for each `SummaryNode` — *how* to extract a scalar value at every report step. The chain has arms for user-defined quantities, block values, aquifer values, region values, global process values, and function-relation well/group/field values. An LGR well node (`LWOPR`, `LWBHP`, etc.) has `node.lgr.has_value() == true` and `node.number == 0`. It is not a block value (`LWOPR` absent from `block_units`). It is not a function-relation (`LWOPR` absent from `funs`). So it falls through to `unknownParameter()` — which allocates no evaluator — and the value is silently absent from every UNSMRY file.

After this PR, a new arm `isLgrWellValue()` fires before `isFunctionRelation()`. It detects LGR well nodes by checking `node.lgr.has_value()` + `node.category == Well` + `lgr_well_units.count(node.keyword)`. When it matches, `lgrWellValue()` constructs an `LgrWellValue` evaluator that (a) verifies the well is in the declared LGR via `Well::get_lgr_well_tag()`, then (b) delegates to the existing `ofun` for the base keyword (`LWOPR` → `WOPR`), producing the same numerical result as the global equivalent.

This PR does **not** add LC* (connection) or LB* (block) evaluators — those are PR-004. It does not change SMSPEC output (PR-002). It does not change SummaryConfig parsing (PR-001). No existing evaluator is modified.

```
SUMMARY deck           SummaryConfig (PR-001)      Factory::create() (this PR)
──────────────         ──────────────────────      ──────────────────────────────
LWOPR              →   SummaryConfigNode            SummaryNode { keyword="LWOPR",
 'LGR2'  'PROD' /        { keyword="LWOPR"    →                   wgname="PROD",
                           lgr_={name="LGR2"} }                   lgr={name="LGR2"} }
                                                     isLgrWellValue() → true
                                                     lgrWellValue()
                                                       LgrWellValue::update()
                                                         get_lgr_well_tag() == "LGR2" ✓
                                                         ofun["WOPR"](fn_args)
                                                         → value in UNSMRY ✓
```

## Change 1 — `Summary.cpp`: `lgr_well_units` map

    { "LWOPT",  Opm::UnitSystem::measure::liquid_surface_volume },

**Key insight:** The unit in this map must match what `funs["WOPR"]` returns (`measure::liquid_surface_rate`). Using a different measure would produce silently wrong numbers — the conversion factor would be wrong while the raw SI value is correct.

## Change 2 — `Summary.cpp`: `base_keyword()` helper

```cpp
static std::string base_keyword(const std::string& lgr_kw)
{
    // Strip leading 'L' from LW*/LC*/LB* keywords.
    // WHY: funs map covers only base keywords (WOPR, WBHP, ...).
    // LWOPR is not a key in funs. Stripping the L gives the correct entry.
    // TRAP: do not strip unconditionally — only strip if first char is 'L'
    // and length > 1, to avoid corrupting any hypothetical 1-char keyword.
    return (lgr_kw.size() > 1 && lgr_kw[0] == 'L')
        ? lgr_kw.substr(1)
        : lgr_kw;
}
```

**Key insight:** This is a file-local static, not a method on `SummaryNode`. The stripping logic is an implementation detail of the evaluator layer — it must not leak into the config or IO layers where the full `LWOPR` name is the canonical identity.

## Change 3 — `Summary.cpp`: `LgrWellValue` evaluator class

```cpp
class LgrWellValue : public Base
{
public:
    void update(...) const override
    {
        const auto& well_name = this->node_.wgname;
        const auto& lgr_name  = this->node_.lgr->name;

        // Guard 1: well must exist at this timestep.
        if (!input.sched.hasWell(well_name, sim_step)) return;

        // Guard 2: well must be in the declared LGR.
        // WHY: get_lgr_well_tag() returns nullopt for WELSPECS wells
        // and the LGR name string for WELSPECL wells.
        // If a different well named "PROD" exists in a different LGR,
        // this guard prevents cross-contamination of values.
        // TRAP: *tag != lgr_name does a string compare — this is correct
        // because lgr_info::name is stored unpadded.
        const auto tag = input.sched.getWell(well_name, sim_step)
                            .get_lgr_well_tag();
        if (!tag.has_value() || *tag != lgr_name) return;

        // Build fn_args exactly as FunctionRelation::update() does.
        // WHY: the ofun lambdas in funs (e.g. rate<rt::oil, producer>)
        // iterate the wells vector to sum rates. If wells is empty,
        // every rate returns zero.
        const auto wells = find_wells(input.sched, this->node_,
                                      static_cast<int>(sim_step), input.reg);
        const fn_args args { wells, ... };

        // Delegate to the base keyword's ofun.
        // NOTE: prm.unit is the measure enum returned by the ofun —
        // we use usys.from_si(prm.unit, prm.value) for unit conversion.
        const auto prm = this->fcn_(args);
        updateValue(this->node_, usys.from_si(prm.unit, prm.value), st);
    }
};
```

**Key insight:** The LGR guard (`get_lgr_well_tag()`) is inside `update()`, not inside `isLgrWellValue()`. The factory check determines *which evaluator class* to use. The per-timestep guard determines *whether to emit a value* at a given step. These are different concerns at different frequencies.

## Change 4 — `Summary.cpp`: `Factory::isLgrWellValue()` + dispatch arm

```cpp
// BEFORE — LGR node falls through:
if (this->isGlobalProcessValue()) return this->globalProcessValue();
// ← LWOPR reaches here, funs.find("LWOPR") fails, falls to unknownParameter()
if (this->isFunctionRelation())   return this->functionRelation();

// AFTER — LGR arm inserted:
if (this->isGlobalProcessValue()) return this->globalProcessValue();
if (this->isLgrWellValue())       return this->lgrWellValue();   // NEW
if (this->isFunctionRelation())   return this->functionRelation();
```

```cpp
bool Factory::isLgrWellValue()
{
    // Three conditions must all be true:
    // 1. Node has an LGR annotation (node_.lgr.has_value())
    // 2. Node is a Well-category node
    // 3. The keyword is in lgr_well_units (known LW* keyword)
    // NOTE: side effects on paramUnit_ and paramFunction_ follow
    // the existing Factory pattern — all is*() methods that match
    // set up the corresponding descriptor fields.
    if (!this->node_->lgr.has_value()) return false;
    if (this->node_->category != Category::Well) return false;
    auto pos = lgr_well_units.find(this->node_->keyword);
    if (pos == lgr_well_units.end()) return false;
    this->paramUnit_ = pos->second;
    // TRAP: normalise_keyword must be called on the BASE keyword ("WOPR"),
    // not the LGR keyword ("LWOPR"). Normalisation may canonicalize
    // case/whitespace — the funs map uses normalised keys.
    const auto normKw = SummaryNode::normalise_keyword(
        Category::Well, base_keyword(this->node_->keyword));
    auto fpos = funs.find(normKw);
    if (fpos == funs.end()) return false;
    this->paramFunction_ = fpos->second;
    return true;
}
```

**Key insight:** `isLgrWellValue()` must fire **before** `isFunctionRelation()`. If placed after, `LWOPR` would fall into `isFunctionRelation()` where `funs.find("LWOPR")` returns `end()` and the node reaches `unknownParameter()` — which is the original bug.

## End State After PR-003

For a deck with `LWOPR / 'LGR2' 'PROD' /`:

```
Config layer (PR-001):
  SummaryConfigNode { keyword="LWOPR", lgr_={name="LGR2", ijk={0,0,0}}, wgname="PROD" }

IO layer (PR-002):
  SMSPEC arrays: KEYWORDS["LWOPR"], WGNAMES["PROD"], LGRS["LGR2"], NUMS[0]

Evaluator (this PR):
  Factory dispatches → LgrWellValue { node={keyword="LWOPR", lgr={name="LGR2"}},
                                      m_=liquid_surface_rate,
                                      fcn_=funs["WOPR"] }
  Per report step:
    get_lgr_well_tag("PROD") == "LGR2" ✓
    ofun["WOPR"](fn_args) → { value=1234.5 [SI], unit=liquid_surface_rate }
    usys.from_si(liquid_surface_rate, 1234.5) → e.g. 777.0 [SM3/day]
    SummaryState["LWOPR:PROD"] = 777.0

UNSMRY:
  LWOPR value written at each report step ✓
  (Was: unknownParameter → no value — the original bug)

Still null until later PRs:
  LC* connection values → PR-004
  LB* block values      → PR-006
```
