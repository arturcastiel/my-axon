## Mental Model

Before this PR, the OPM summary pipeline had no way to carry LGR identity from the
input parser to the binary SMSPEC/UNSMRY writers. The `SummaryConfigNode` class
(config layer) was missing an LGR field entirely. The IO-layer struct `EclIO::SummaryNode`
already had an `optional<lgr_info> lgr` field, but the conversion
`operator EclIO::SummaryNode()` hard-coded `lgr = {}` — so LGR identity was silently
discarded every time a node crossed the layer boundary. Additionally,
`category_from_keyword()` had no `case 'L':`, so `LWOPR` / `LCOFR` / `LBPR` were all
classified as `Miscellaneous` — broken at the very first step.

This PR makes four isolated, purely additive fixes. None touch parsing logic, evaluator
logic, or SMSPEC writing. They establish the data plumbing every subsequent PR relies on.

**Before:** any keyword starting with `L` → `Miscellaneous` category, no LGR field in
`SummaryConfigNode`, conversion to `EclIO::SummaryNode` hard-codes `lgr = {}`.

**After:** `LW*` → `Well`, `LC*` → `Connection`, `LB*` → `Block`. `SummaryConfigNode`
carries `optional<string> lgr_name_` and `optional<size_t> lgr_index_`. Conversion
passes LGR name into `EclIO::lgr_info`. Deduplication and sorting work correctly on LGR
nodes.

**What this PR does NOT do:** it does not parse any LW*/LC*/LB* keywords (that is
PR-001), does not write any SMSPEC arrays (PR-002), and does not add any evaluator
(PR-003). It is purely a data structure and classification fix.

```
SUMMARY deck
     │
     ▼
category_from_keyword()          ← SummaryNode.cpp
  case 'L': [NEW]
  LW* → Well, LC* → Connection, LB* → Block
     │
     ▼
SummaryConfigNode                ← SummaryConfig.hpp
  lgr_name_  optional<string>    [NEW field]

     │

EclIO::SummaryNode               ← SummaryNode.hpp
  lgr  optional<lgr_info>        [EXISTED — now populated instead of hard-coded {}]
     │
     ▼
SMSPEC / UNSMRY (PR-002+)
```

## Key Disambiguation

Two classification functions exist and are **completely separate**. Both need `case 'L':`,
but they live in different files and are fixed in different PRs.

| Function | File | Layer | PR |
|---|---|---|---|
| `SummaryNode::category_from_keyword(kw)` | `SummaryNode.cpp` | EclIO (IO layer) | **PR-000** |
| `parseKeywordCategory(kw)` | `SummaryConfig.cpp` | Input (config layer) | **PR-001** |

`category_from_keyword` is a static method on `EclIO::SummaryNode` — it classifies a
keyword string for the binary output layer. `parseKeywordCategory` is a free function in
the config layer used during deck parsing. They are independent. Adding `case 'L':` to
only one of them without the other would break either the IO or the parsing side.
**PR-000 fixes the IO-layer one; PR-001 fixes the config-layer one.**

## Change 1 — `SummaryNode.cpp`: Add `case 'L':` to `category_from_keyword()`

**Before:**
```cpp
switch (keyword[0]) {
case 'A': return Category::Aquifer;
case 'B': return Category::Block;
case 'C': return distinguish_connection_from_completion(keyword);
case 'F': return Category::Field;
case 'G': return distinguish_group_from_node(keyword);
case 'R': return Category::Region;     // L falls through to default
case 'S': return Category::Segment;
case 'W': return distinguish_well_from_completion(keyword);
default:  return Category::Miscellaneous;  // ← LWOPR lands here — wrong
}
```

**After:**
```cpp
case 'L':
    // WHY nested switch: LW*/LC*/LB* follow the same sub-letter
    // pattern as W*/C*/B* at the global level.
    if (keyword.size() >= 2) {
        switch (keyword[1]) {
        case 'W': return distinguish_well_from_completion(keyword);
        // WHY reuse distinguish_well_from_completion: this helper
        // contains a regex for well-completion suffixes (WOPRL etc).
        // For LW* keywords: keyword[0]='L' so the regex (which requires
        // 'W' at position 0) never matches → always returns Well.
        // This is correct AND future-proof for any LWOPRL variants.
        case 'C': return distinguish_connection_from_completion(keyword);
        // WHY same reasoning: regex requires 'C' at position 0 → never
        // matches LCxxx → always returns Connection.
        case 'B': return Category::Block;
        // WHY direct: no completion variant for LB* keywords exists.
        default:  break;
        // NOTE: LLINEARS has keyword[1]='L' → falls through here
        // → outer return Category::Miscellaneous. Correct.
        }
    }
    return Category::Miscellaneous;
    // NOTE: inserted before 'R' to keep alphabetical order in the switch.
```

**Key insight:** The `distinguish_*` helpers are safe to reuse for `LW*`/`LC*` because
their internal regexes anchor on the first character (`W` or `C`). Since `LWOPR` starts
with `L`, the regex never matches, so the helper always returns `Well` — the correct
result.

## Change 2 — `SummaryNode.hpp`: Add `operator==` and `serializeOp` to `lgr_info`

This change is required for the config layer to use `optional<lgr_info>` in the IO
layer's `EclIO::SummaryNode`. It is also needed by `SummaryNode.hpp` itself for
downstream equality comparisons.

**Before:**
```cpp
struct lgr_info {
    std::string       name {};
    std::array<int,3> ijk  {};
    // no operator==, no serializeOp
};
```

**After:**
```cpp
struct lgr_info {
    std::string       name {};
    std::array<int,3> ijk  {};

    bool operator==(const lgr_info&) const = default;
    // WHY =default: both member types (string, array<int,3>) support
    // memberwise equality in C++20. OPM targets C++20 — confirmed by
    // use of std::ranges throughout.
    // WHY needed at all: std::optional<lgr_info>::operator== requires
    // lgr_info::operator==. Without this, the operator== and operator<
    // fixes in SummaryConfig.cpp fail to compile.

    template<class Serializer>
    void serializeOp(Serializer& s)
    {
        s(name);
        s(ijk);
        // WHY both: the OPM Serializer calls serializeOp on any type
        // that has it. string is handled directly; array<int,3> is
        // handled via the array branch (Serializer.hpp:122-123).
        // Without this, serializer(lgr_) in SummaryConfig.hpp fails
        // to compile because the packer cannot reach the string member.
    }
};
```

**Key insight:** These two methods are required purely to make other code compile —
they have no runtime behaviour change at this PR stage. `operator==` enables optional
comparison; `serializeOp` enables the OPM serializer to pack the struct.

## Change 3 — `SummaryConfig.hpp`: Replace `optional<lgr_info>` with `lgr_name_` + `lgr_index_`

This is the central change of the revised design. The original implementation stored
`optional<EclIO::lgr_info>` (name + ijk array) in `SummaryConfigNode`. Reviewer @bska
objected to:
1. The ijk array — it belongs in the IO layer, not the config layer
2. `array<>::operator<` — lexicographic ordering on ijk is not declaration order

**Revised design: two separate optional fields.**

### 3a — Private fields

**Remove:**
```cpp
std::optional<Opm::EclIO::lgr_info> lgr_{};
```

**Add:**
```cpp
std::optional<std::string>   lgr_name_{};
// WHY string not index: the name comes directly from the deck record
// at parse time. No grid object is needed to get it.

std::optional<std::size_t>   lgr_index_{};
// WHY size_t: this is the 0-based LGR index from
// EclipseGrid::get_lgr_cell_index(name). It gives monotonically
// increasing declaration-order integers (LGR1=0, LGR2=1, ...).
// Used as the sort key in operator< — clean, no array ordering.
// NOTE: set in PR-001 by keywordLW/LC/LB if grid index is available.
// May remain nullopt at parse time (GridDims has no LGR labels).
// In that case operator< falls back to lgr_name_ string comparison.
```

### 3b — Public getters and setters

```cpp
const std::optional<std::string>& lgr_name() const
{ return this->lgr_name_; }

SummaryConfigNode& lgr_name(const std::string& name)
{ this->lgr_name_ = name; return *this; }
// WHY fluent (return *this): consistent with all other setters on
// this class (namedEntity, number, parameterType, etc.).

const std::optional<std::size_t>& lgr_index() const
{ return this->lgr_index_; }

SummaryConfigNode& lgr_index(std::size_t num)
{ this->lgr_index_ = num; return *this; }
```

### 3c — Fix `operator EclIO::SummaryNode()` conversion

**Before:**
```cpp
/* lgr = */ {}  // hard-coded empty optional — LGR identity always lost
```

**After:**
```cpp
/* lgr = */ this->lgr_name_.has_value()
            ? std::optional<EclIO::lgr_info>{ EclIO::lgr_info{ *this->lgr_name_, {0,0,0} } }
            : std::nullopt
// WHY ijk = {0,0,0}: the ijk coords are not stored in SummaryConfigNode
// in the revised design. For LW*/LC* nodes, ijk is not meaningful at
// the config-layer level (no specific cell). For LB* nodes the active
// cell index is in number_ and is resolved by the evaluator.
// The IO layer (OutputStream.cpp) writes NUMLX/Y/Z from lgr->ijk —
// these will be 0 for LW*/LC*, which is correct for those categories.
```

**Key insight:** `lgr_name_` alone is sufficient to populate the IO-layer `lgr` field.
The ijk array in `lgr_info` is used by `OutputStream.cpp` for NUMLX/Y/Z — but for
LW*/LC* vectors those values are legitimately 0 (no specific cell). LB* uses `number_`
for the active cell index, not ijk.

### 3d — Fix `serializeOp`

**Remove:**
```cpp
serializer(lgr_);
```

**Add (at the end — binary layout constraint):**
```cpp
serializer(lgr_name_);
serializer(lgr_index_);
// WHY at the end: the OPM binary serializer writes fields in the order
// serializeOp calls them. Adding new fields LAST means existing
// serialized streams (restart files without LGR data) remain valid —
// they simply have nothing at the end. Inserting mid-stream would
// corrupt every existing restart file.
// TRAP: changing the order of these two lines also breaks compatibility.
```

## Change 4 — `SummaryConfig.cpp`: Fix `operator==`, `operator<`, `serializationTestObject`

### Fix `operator==`

**Remove** (Well/Node/Group case):
```cpp
return lhs.namedEntity() == rhs.namedEntity();
```
**Add:**
```cpp
return (lhs.namedEntity() == rhs.namedEntity())
    && (lhs.lgr_name()    == rhs.lgr_name());
// WHY lgr_name_ not lgr_index_: name is always set when lgr is
// present. lgr_index_ may be nullopt. Two nodes with the same name
// in the same LGR must always compare equal, regardless of whether
// lgr_index_ was resolved.
```

Same pattern for Block/Aquifer/Region (add `&& lhs.lgr_name() == rhs.lgr_name()`)
and Connection/Completion/Segment (add the same).

**Why this matters:** `SummaryConfig::uniq()` deduplicates the node list using
`operator==`. Without `lgr_name_` in the comparison, `LWOPR` for `PROD1` in `LGR1`
and `LWOPR` for `PROD1` in `LGR2` compare equal and one is silently dropped.

### Fix `operator<`

**Old tiebreaker (Well case):**
```cpp
return lhs.namedEntity() < rhs.namedEntity();
```

**New tiebreaker:**
```cpp
if (lhs.namedEntity() != rhs.namedEntity())
    return lhs.namedEntity() < rhs.namedEntity();

// Same named entity — break tie on LGR
const auto& ln = lhs.lgr_name();
const auto& rn = rhs.lgr_name();
if (ln.has_value() != rn.has_value())
    return rn.has_value();  // no-LGR sorts before LGR
    // WHY: nodes without LGR should appear first in sorted output,
    // consistent with how non-LGR vectors appear before LGR variants.
if (!ln.has_value())
    return false;           // both null → equal

// Both have LGR — use lgr_index_ if available (declaration order)
const auto& lnum = lhs.lgr_index();
const auto& rnum = rhs.lgr_index();
if (lnum.has_value() && rnum.has_value() && (*lnum != *rnum))
    return *lnum < *rnum;
    // WHY lgr_index_: gives monotonically increasing declaration-order
    // integers (LGR1=0, LGR2=1). This is what reviewer @bska wanted —
    // avoids array<>::operator< which gives {1,2,3} < {2,2,2} (index
    // tuple ordering, not declaration order).

// Fall back to name comparison if numbers not available
return *ln < *rn;
// NOTE: both operators (== and <) use lgr_name_ as the authority.
// lgr_index_ is a tie-breaking optimisation only.
```

Apply the same pattern to Block and Connection cases.

### Fix `serializationTestObject`

**Remove:**
```cpp
result.lgr_ = EclIO::lgr_info{ "LGR1", {1, 2, 3} };
```
**Add:**
```cpp
result.lgr_name_   = std::string{ "LGR1" };
result.lgr_index_ = std::size_t{ 1 };
// WHY: serializationTestObject must exercise every field so the
// binary roundtrip test catches regressions in serializeOp.
// Both new fields must be set to non-default values here.
```

### Recurring Traps

- **`BOOST_CHECK_EQUAL` on `Category` enum:** `Category` has no `operator<<` — use
  `BOOST_CHECK(a == Category::Well)` not `BOOST_CHECK_EQUAL(a, Category::Well)`.
  The latter fails at compile time with a confusing error about missing `<<`.
- **Checking `lgr_name_` vs `lgr_` in tests after the change:** all test code that
  previously called `.lgr()` must be updated to `.lgr_name()`. A build error will
  catch this immediately.
