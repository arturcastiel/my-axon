## G-LIB-001 — library-dev shadow missing metadata enrichment

**type:** absent-instruction
**detected-by:** user audit (shadow metadata completeness review)
**program(s):** `library-dev-ingest.md`

**Gap:**
`library-dev-ingest` constructs shadow notes via `DERIVE()` + `WRITE()` inline — it does not call `TOOL(shadow, init)`.
As a result, library shadow notes carry **no git metadata**:
- no `git-branch`
- no `git-commit`
- no `--caller-program`
- no `--caller-project`
- no branch-staleness detection at explain/search time

**Expected behavior:**
Library shadow notes should record the same metadata that `code-dev-shadow` and `code-dev-study` pass:
- `--branch {git-branch}` — branch of the library's source repo at ingest time
- `--commit {git-commit}` — HEAD hash at ingest time
- `--caller-program library-dev-ingest`
- `--caller-project {W:library-dev-active}` (library name as project ID)

**Context:**
Library shadows are static article notes (PDFs/TXTs), not source files — they don't change with branches. However, recording the ingest branch/commit allows:
- knowing *when* the library was ingested relative to codebase state
- potential future branch-stale detection (e.g., if library is associated with a codebase repo)

**Proposed fix:**
Option A — Refactor ingest to call `TOOL(shadow, init, ...)` instead of bare `WRITE()`.
Option B — Append metadata header block to the `WRITE()` call (simpler, no shadow-tool dependency).
Option C — Add `--caller-program` and `--caller-project` only; skip git metadata (libraries are not tied to a git repo).

**Recommendation:** Option C first (low-effort, unblocks caller tracing); Option A/B for full git metadata if library-dev gets associated-repo support.

**status:** logged — pending planning

## G-DEV-002 — incomplete ctest command: SummaryConfigNode (#77) excluded

**type:** absent-verification
**detected-by:** self-audit during review investigation
**session:** 2026-05-07 opm-common PR-001 session

**Gap:**
The ctest regex `^(SummaryNode|SummaryConfigTests|Serialization)$` was used to validate PR-001 throughout the session. Test target `SummaryConfigNode` (#77) — which runs `test_SummaryConfigNode.cpp` and contains ALL new PR-001 suites (`ParseKeywords/Category`, `NoSumLgr`, `LGR_SummaryConfig_Deck`, `LGR_Schema_Inline`) — was not included in the regex.

As a result:
- New suites were compiled into a binary that was never run via ctest
- `SummaryConfigTests` (#156) runs a different file (`tests/parser/SummaryConfigTests.cpp`) — not the file we modified
- The binary for `SummaryConfigNode` was stale (showed only 3 old PR-000 cases)
- Session reported "all tests passed" — this was incorrect; the new tests were not run at all

**Correct ctest command:**
```
ctest -R "^(SummaryNode|SummaryConfigNode|SummaryConfigTests|Serialization)$" --output-on-failure
```

**Impact:**
Medium — PR-001 was committed and almost pushed without running its own test suites. Potential real test failures (BLOCKER-3/4/5 from the unsolicited review) remain unverified.

**Prevention:**
After adding tests to a file, always verify which ctest target number corresponds to that file (`ctest -N | grep <target-name>`). Establish the correct target list at the start of a PR session and do not rely on assumed naming.

**status:** logged — pending: rebuild test binary and run correct ctest command before push
